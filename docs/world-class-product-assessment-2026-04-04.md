## 1. 总体结论

- 当前项目已经明显超过原型期，不是“演示拼装品”，而是一个具备真实教育产品主线、可进入受控试点的强 beta 产品。
- 项目最强的部分不是单个页面，而是产品方向本身：教师授课、学生自学、数字人、整班观看、课堂回看、导出沉淀、教务角色打通，这条线非常有价值，也具备差异化潜力。
- 同日新增的 P1/P2 能力交付已经基本落到真实代码与自动化回归里，说明项目正在从“概念正确”进入“可维护、可复验”的产品化阶段。
- 当前离“世界级产品”的差距，主要不在“功能缺不缺”，而在四个收口层：
  - 核心链路是否彻底服务端化
  - 设计系统是否真正沉淀为统一产品语法
  - 多角色黄金路径是否被压缩到足够清晰
  - 测试与运行恢复体系是否足以承接真实学校环境
- 如果目标是 `单校 / 少量学校试点`，项目可继续推进。
- 如果目标是 `品牌级对外展示 + 多学校复制 + 高稳定性交付`，还需要一轮明确的 P0 / P1 / P2 产品化改造。

建议定级：

- 世界级产品潜力：`高`
- 当前产品成熟度：`中高`
- 当前可落地成熟度：`中`
- 当前世界级完成度：`中`

## 3. 已具备的强项

### 3.1 产品方向是对的

- 项目不是围绕单点 AI 功能，而是围绕“教学活动”本身组织能力。
- 互动课堂不只是教师演示工具，还开始承载：
  - 学生预习
  - 学生学科巩固
  - 兴趣培养
  - 课堂回看
  - 课后继续练习
- 这种“同一课堂资产既可授课又可自主学习”的产品方向，本身就具备世界级产品的潜力。

### 3.2 工程基础明显高于一般 demo

- 仓库当前体量已经不小：
  - `79` 个页面
  - `237` 个 API 路由
  - `109` 个单测文件
  - `360` 条单测
  - `2` 个浏览器自动化文件
  - `21` 条浏览器自动化用例
- 已具备：
  - `build`
  - `unit test`
  - `api test`
  - `browser smoke`
  - `launch readiness`
  - realistic seed / production-like runbook
- 2026-04-04 已通过 `corepack pnpm check:project-snapshot && corepack pnpm verify:strict`，说明最新一轮能力扩展不是“只在文档里完成”。
- 这说明项目已经在向“可交付工程”靠近，而不是只做前台展示。

### 3.3 治理意识已经进入代码，而不是停留在口号

- 数据库强依赖与运行时 guardrail 已建立，见 `lib/db.ts`
- Provider Vault 与服务端 Provider 配置体系已建立，见 `lib/server/provider-config.ts`
- Provider secret policy 已具备环境级拦截逻辑，见 `lib/server/provider-secret-policy.ts`
- 这意味着项目不是“无控制地堆 AI 功能”，而是已经有明确的治理骨架。

### 3.4 最近一轮 UI 收口方向正确

- 学生自主学习页的首屏已经明显比之前更清爽。
- `generation-preview` 的舞台比例、说明层级与等待期前厅，已经开始从“控制台”往“产品主舞台”转。
- 互动课堂相关页面已经开始形成更统一的教育产品语气，而不是技术工具语气。

## 4. 最高优先级问题

以下问题按“如果目标是世界级产品，必须优先解决”的顺序排列。

### P1. 核心 AI 生成链路仍未彻底服务端化

当前问题：

- 生成预览页仍会在浏览器侧读取配置并拼装请求头，把模型、密钥、Base URL 等信息带入请求。
- 关键证据：
  - `app/generation-preview/page.tsx`
  - `lib/server/resolve-model.ts`
  - `app/api/generate/image/route.ts`
  - `app/api/generate/video/route.ts`
  - `lib/server/provider-secret-policy.ts`

本质风险：

- 密钥治理口径不够彻底
- 真实生产与本地调试存在双路径心智
- 任务审计、任务重放、故障归因、租户隔离和成本控制都更难标准化

世界级标准下应达到：

- 客户端只提交任务意图和资源引用
- 服务端统一做：
  - provider 选择
  - 密钥解析
  - 模型路由
  - 成本预算
  - 审计与告警

结论：这是当前最重要的工程收口点。

### P1. 生成会话仍过度依赖浏览器态，缺少真正的恢复力

当前问题：

- 课堂生成会话主要写入 `sessionStorage`
- 生成预览页从浏览器态恢复并自动继续执行
- 关键证据：
  - `app/ai-classroom/page.tsx`
  - `app/generation-preview/page.tsx`

本质风险：

- 刷新、切设备、切浏览器、切账号时恢复能力不完整
- 老师备课和学生续学无法形成统一可审计记录
- 任务级异步编排与后台重试体系很难建立

世界级标准下应达到：

- 生成会话是服务端 job / workflow
- 浏览器只作为观察者和控制器
- 可支持：
  - 异步恢复
  - 多端继续
  - 失败重试
  - 管理台追踪

结论：这是从“强原型”升级到“可复制产品”的关键分水岭。

### P1. 核心页面仍过大，后续扩展会越来越慢

当前问题：

- `app/student/interactive-classroom/page.tsx`：`2296` 行
- `app/generation-preview/page.tsx`：`1952` 行
- `app/ai-classroom/page.tsx`：`2268` 行
- `app/classroom/[id]/page.tsx`：`930` 行

本质风险：

- 单文件同时承担：
  - 产品编排
  - 状态管理
  - 数据准备
  - 错误处理
  - UI 结构
  - 视觉细节
- 功能越叠越容易影响回归范围
- 新人接手和团队协作成本会显著上升

世界级标准下应达到：

- 页面只做 page composition
- 中层拆成：
  - loaders
  - actions
  - page state
  - orchestration hooks
  - visual sections
  - pure derived logic

结论：这是影响研发速度和质量门的长期风险点。

## 5. UI / UX 评估

### 5.1 当前已经做对的地方

- 整体视觉已经从“普通后台”转向“教育产品”。
- 颜色与氛围更明亮，摆脱了此前偏灰、偏重、偏工具态的问题。
- 互动课堂相关页面已经有明显的“主舞台”意识。
- 学生自主学习页开始体现“学习工作台”而不是“表单页”。

### 5.2 距世界级还差的部分

#### A. 首屏仍然有“说明过多”的惯性

典型页面：

- `app/student/interactive-classroom/page.tsx`
- `app/generation-preview/page.tsx`

当前问题不是乱，而是：

- 首屏在同时解释很多事
- 用户要先读结构，再决定动作
- 一些页面已经开始收口，但还没达到“第一眼就知道下一步做什么”的程度

世界级产品通常会做到：

- 首屏只承载一个主决定
- 次要信息后置
- 关键路径永远最亮、最短、最不犹豫

#### B. 页面状态间的“人格一致性”还不够

例如 `generation-preview`：

- 无会话态像入口页
- 生成中态像工作台
- 恢复态又像系统状态页

这会让用户感到“这不是同一空间的不同状态”，而像“跳了不同页面风格”。

世界级标准下要做到：

- 同一页面在不同状态下仍保留统一语法
- 同一品牌、同一信息层级、同一行为预期

#### C. 设计系统还没有完全产品化

从页面中仍可看到不少内联样式与局部结构组合，典型页：

- `app/student/interactive-classroom/page.tsx`
- `app/classroom/[id]/page.tsx`

这意味着当前视觉统一更多依赖“人工细调”，而不是“设计系统天然保证”。

世界级标准下应建立统一 section schema：

- hero
- focus card
- mode card
- action rail
- summary card
- disclosure block
- stage shell
- empty / loading / error state shell

#### D. 全屏沉浸式课堂还没有完全成为第一优先级

你已经明确提出“支持全屏播放无覆盖”，这是非常正确的方向。

按世界级标准，课堂播放页应该做到：

- chrome 最小化
- 课堂内容绝对中心
- 所有辅助信息按需出现
- 老师投屏和学生观看时都不被杂项打断

当前方向已经对，但还需要继续把“播放页”从“功能页”提升为“观课空间”。

## 6. 用户体验评估

### 6.1 教师体验

优点：

- 教师端功能深度足够
- 互动课堂、AI 工具、模块、通知、分析等已经不是空壳
- 课堂生成与教务角色结合方向明确

风险：

- 老师最需要的是“低负担开课”，而不是“理解系统很强”
- 若配置、生成、导出、回看、继续教学这些环节不够丝滑，老师会迅速觉得重

建议：

- 老师黄金路径压缩成：
  - 选班级 / 选主题
  - 一键生成
  - 预览确认
  - 进入课堂 / 导出 / 分享

### 6.2 学生体验

优点：

- 学生自主学习已经有真正的产品意义，不再只是“陪聊 AI”
- 预习 / 巩固 / 兴趣 / 回看四模式的结构是对的
- 可以继续深挖成世界级差异化能力

风险：

- 学生端仍有部分页面信息量偏大
- 首次用户有时需要先理解结构，再敢行动
- 需要进一步压缩为“轻决策、快进入、强反馈”

建议：

- 学生端永远优先：
  - 我现在学什么
  - 为什么学这个
  - 点哪里开始
  - 结束后得到什么

### 6.3 学校与管理端体验

优点：

- 学校端、管理端已具备真实治理能力，不只是展示面板
- Provider Vault、模型路由、launch readiness 都很有产品价值

风险：

- 管理能力多，但“运营视角的一体化叙事”还不够强
- 部分能力更像“工程能力存在”，而不是“管理者一眼明白价值”

建议：

- 学校端要更强调：
  - 已开课规模
  - 已导出与已回看规模
  - 教师使用率
  - 学生完成率
  - AI 课堂对学习结果的实际贡献

## 7. 技术与工程评估

### 7.1 优点

- `Next.js App Router + app/api + lib/*` 的结构已经形成稳定主骨架
- DB-only guardrail 思路明确
- realistic seed、launch readiness、production-like smoke 这些都很有价值
- 有较强的单测意识和回归意识

### 7.2 关键短板

#### A. 浏览器端 E2E 覆盖度不够

当前 `tests/browser` 目录只有 `1` 个文件：`tests/browser/smoke.spec.ts`

这对当前项目体量来说是不够的。

项目已经覆盖：

- 学生
- 教师
- 家长
- 学校
- 管理员
- 互动课堂
- 自主学习
- 资料库
- 排课
- 恢复流程

世界级标准下，E2E 应至少拆成：

- auth / security
- student self-study
- teacher classroom launch
- generation preview & recovery
- classroom playback & fullscreen
- export & replay
- admin provider governance
- school oversight

#### B. 任务编排与 UI 状态还耦合太深

当前生成链路很多逻辑仍然停留在页面内 orchestrate。

世界级标准下应拆为：

- task submission
- task runner
- progress polling / streaming
- recovery / retry
- UI projection

#### C. 设计系统与业务组件还没有彻底分层

已有大量复用卡片，但还不够“系统”。

建议把课堂相关组件分成：

- `product-shell`
- `education-sections`
- `classroom-stage-system`
- `role-aware action patterns`

#### D. 运行恢复、灰度、容错还需要再产品化

当前已经有：

- readiness
- production-like runbook
- provider policy

但还需要进一步把这些能力前台化、流程化，而不是停留在工程侧工具。

## 8. 安全评估

### 8.1 当前优点

- 数据库 guardrail 明确
- provider vault 已存在
- provider secret override 已有策略拦截
- SSRF 校验已覆盖部分外部 provider 链路

### 8.2 当前最需要继续收口的点

- 客户端真实密钥参与链路必须继续压缩，直到只保留服务端托管路径
- 生成、媒体、数字人相关任务统一纳入服务端审计
- 多租户、角色边界、班级边界要继续保持强回归测试
- 管理端高风险动作建议进一步强化“二次确认 + 审计留痕 + 回滚说明”

结论：

- 安全不是弱项，但也还没有达到“可放心作为世界级产品品牌背书”的程度

## 9. 世界级标准下的目标形态

如果把这个产品做到世界级，应该长成这样：

### 9.1 产品层

- 老师能 3 步发起一节真实可用的 AI 互动课堂
- 学生能 1 分钟内进入自主学习
- 同一课堂资产可授课、可回看、可导出、可继续追练
- 学校能看到“开课 -> 使用 -> 结果”的统一证据链

### 9.2 体验层

- 首屏不解释系统，而是直接推动行动
- 课堂播放页足够沉浸，能投屏、能全屏、能观课
- 整体视觉克制、明亮、有教育品牌感，而不是后台工具感

### 9.3 工程层

- 核心生成链路全部服务端任务化
- 配置与密钥全部后台统一管理
- E2E 测试覆盖关键角色矩阵
- 故障、重试、回滚、审计全部可追踪

## 10. 改造建议路线图

## P0：必须先做

1. 核心生成链路服务端化
   - 生成会话、媒体生成、数字人生成、导出统一改成服务端任务
   - 客户端不再持有真实 provider 密钥

2. 互动课堂主链路拆层
   - 拆 `ai-classroom`
   - 拆 `generation-preview`
   - 拆 `student/interactive-classroom`
   - 拆 `classroom/[id]`

3. 课堂播放体验收口
   - 全屏播放
   - 最小化遮挡
   - 统一播放控制层
   - 观课 / 投屏 / 学生自学三种视角统一

4. 浏览器回归矩阵扩容
   - 至少补齐 6 到 8 个 E2E 主题套件

## P1：试点到品牌化的关键一步

1. 设计系统化
   - 提炼互动课堂 section schema
   - 建立统一 spacing / card / stage / disclosure / status tokens

2. 学生自主学习黄金路径再压缩
   - 首次进入更短
   - 开课前设置更轻
   - 课后回流更清晰

3. 学校端结果看板升级
   - 从“管理能力列表”升级成“效果看板”

4. 服务端任务与管理台追踪接通
   - 让管理者能看见：
     - 谁发起
     - 用了什么模型
     - 生成是否成功
     - 导出是否完成
     - 哪些班级 / 学生在使用

## P2：世界级差异化

1. 学生长期学习画像
2. 数字人角色资产中心
3. 更强的课堂回放与成果沉淀
4. 学校级品牌化和租户运营能力

## 12. 最终判断

这不是一个“还得先证明有没有价值”的项目。

这已经是一个方向很强、产品潜力很高、教育场景非常值得深挖的 AI 教育产品雏形成品。真正需要做的，不是继续证明“这个方向值不值得”，而是把它从“强 beta”推进到“世界级可落地产品”。

当前最关键的一句话总结：

> 项目最需要的不是再加更多功能，而是把已经很强的产品主线，通过服务端化、设计系统化、黄金路径收口和质量门升级，真正推到可规模复制的产品级。
